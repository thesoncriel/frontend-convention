# PR Code Review

작성자의 PR 리뷰 요청은 불특정 다수에겐 슬랙방을, 특정 1인에게는 DM 혹은 구두로 요청합니다.

특별한 사유가 없다면 **리뷰어는 가급적 요청받은 PR 리뷰는 당일안에 해결**합니다.

(만약 부득이하게 당일내에 리뷰가 불가하다면, 사유와 리뷰 가능한 일시를 해당 PR에 코멘트로 남깁니다.)

본 문서는 리뷰어와 작성자간의 코드 리뷰중 발생될 수 있는 오해와 갈등을 미연에 방지하기 위한 방법을 서술합니다.

아래 내용은 예시에 따른 **권장사항**이며 상호간에 더 나은 피드백 방법이 있다면 그 것을 지향해도 좋습니다.

## 리뷰 피드백 유의사항

리뷰어가 코드를 리뷰하며 댓글을 달 때와

작성자가 리뷰어의 댓글에 대해 피드백을 줄 때.

이 2가지 경우 모두 `상호 배려와 존중`이 필요합니다.

가령 댓글을 다는 경우 다음과 같은 상황에 유의합니다.

```ts
// 지목된 코드. 하나라도 isValid 가 true 라면 순회를 돌지 않게 하는게 목적.
let finished = false;

items.map(item => {
  if (finished) {
    return;
  }
  if (item.isValid) {
    finished = true;
    return;
  }
  item.error = getErrorMessage(item.key);

  return;
});
```
```
-- bad
이 부분은 map 이 아닌 some 를 쓰세요.

-- good
제가 보기엔 map 이 아닌 some 를 사용하는게 좋을 것 같습니다.
사유는 map 은 배열 요소를 모두 순환하며 데이터를 변환하는게 목적인 반면에
여기서는 `item.isValid` 가 true 일때 더이상 순회하지 않기를 원하는 의도가 보였기 때문입니다.
만약 다른 의도가 있었다면 알려주실 수 있을까요? 😄
```

즉 단답형 및 명령조 보다는 `나의 생각을 제안하는 방향`으로의 작성을 권합니다.

또한 아래와 같은 상황에 유의합니다.

```ts
// 지목된 코드. 옷상품에서 특정 이벤트 명칭에 맞는 3가지 자료를 추출해서 전달한다.
function findItem(items: ClothItemUiState[], name: string) {
  return items.find(item => item.eventName === name);
}

function convertToEventDataFrom(items: ClothItemUiState[]) {
  const eventItem = findItem(items, '이벤트');
  const summerItem = findItem(items, '여름');
  const chuseokItem = findItem(items, '추석');

  return {
    eventItem,
    summerItem,
    chuseokItem,
  };
}
```

```
-- bad
theson 님, 이걸 이렇게 짜시면 convertToEventDataFrom 함수 수행 때 마다 옷상품 자료를 매번 순회하게 되는데 normalize 하셔야 되지 않을까요?

-- good
convertToEventDataFrom 함수에서 순회가 너무 잦은 것 같습니다!
그에 따른 성능 저하가 우려되는데, 괜찮으시다면 normalize 하도록 로직을 바꿔 보시는건 어떨까요?!
만약 이렇게 작성하신 사유가 있다면 함께 알려주시면 좋을거 같아요! 🙏
```

상기와 같이 `작성자나 리뷰어를 직접적으로 언급하지 않는 것`이 좋습니다.

댓글을 단 사람은 의도하지 않았더라도 문장의 어조에 따라서 받아들이는 바가 달라질 수 있기 때문입니다.

> ℹ️ 팁
> 
> 서로 오고 가는 리뷰글에 이모지나 이모티콘을 활용하면 좋습니다!
>
> 온라인에 남기는 글들은 사뭇 딱딱해질 수 있기에 자칫 오해할 수 있기 때문입니다.
>
> 이러한 오해는 여러 시각적인 요소들을 활용하여 미연에 방지할 수 있습니다.

## 변경 제안 받아들이기

작성자는 리뷰어의 질문에 가급적 답변 해 주어야 합니다.

리뷰어는 단순 질문이 아닌 변경 제안이라면 가급적 제안한 사유와 근거도 함께 작성 해 주십시요.

제안된 내용이 충분히 납득이 된다면 이 역시 가급적 작성자는 수용해 주십시요.

납득이 안된다면 리뷰어에게 보충 설명을 부탁하여 상호 의견이 이해가 되도록 노력해야 합니다.

### 댓글 핑퐁: 납득이 안되거나 사유에 대한 설명이 힘들때

코드 리뷰에 대한 댓글이 서로 납득되지 못하여 `3번 이상 논쟁`이 오고가면 댓글 리뷰는 중단하고 `오프라인 코드리뷰를 권장`합니다.

만약 대상자 중 한명이 원격 근무중이라면 화상회의를 해서라도 서로 이해하고 넘어 가시는걸 권장 합니다.

```
-- bad
A: 이건 Observer 가 아니라 Strategy 를 써야 맞는거 아닌가요?
B: 여기엔 Strategy 가 맞다고 생각해서 적용 했습니다.
A: Strategy 와 Observer 차이는 알고 계신가요?
B: ...(답 없음)

-- good
A: 제가 보기엔 지속적인 콜백 호출을 필요로 하는 게 아니라 콜백의 결과를 원하는 같습니다!
  제 생각엔 Observer 가 아니라 Strategy 를 적용하는게 좀 더 알맞아 보입니다만,
  이 코드에 제가 생각치 못한 다른 사연이 있다면 알려주실 수 있을까요? 😄
B: 아, 넵. 확인 해 보니 Strategy 가 맞는 것 같습니다.
  다만 한 군데가 Observer 형식이 필요한 곳이 있어서 일단 저렇게 두었습니다. 🤔
A: 아, 그러시군요! 시간 되신다면 Observer 가 필요했던 부분을 함께 리뷰 해주실 수 있을까요?
  확인하고 좀 더 좋은 아이디어가 있다면 같이 반영 해 보고 싶습니다! 🙇‍♂️
B: 오~ 좋습니다! 👍
```

상기와 같이 감정적인 어조보다는 상대편의 상황을 이해하기 위한 어조로 작성하길 권합니다.

그리고 서로 얘기가 오고 가는게 길어질 것 같으면 과감히 오프라인 리뷰를 진행하세요!


> ℹ️ 잊지마세요!
> 
> 코드 리뷰는 내가 옳음을 증명하는 자리가 아니라 서로의 생각이 다름을 인정하고 서로가 이해하고 소통하는 자리 입니다!

## 규칙의 의의

함께 하는 개발 업무에 있어 코드 리뷰는 필연적입니다.

이러한 문제를 상호 존중하며 소통하고 서로가 몰랐던 것을 배워가는 장소로 탈바꿈 하여 해결 할 수 있습니다.

이를 통하여 더욱 즐거운 개발 업무 환경이 조성 될 수 있습니다.
