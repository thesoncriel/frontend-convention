# Naming: Custom Type

개발 업무중 여러가지 사용자 정의 타입(이하 커스텀 타입)을 만들어 쓰게 됩니다.

이들에 대한 기본적인 명명 규칙을 이곳에 기재 해 두니 숙지하여 업무에 반영 시켜봅시다.

> ℹ️ 팁
>
> Entity 와 UI State 는 [타입 선언](../architecture/type_definition.md) 문서에서 다루고 있으므로 참고 바랍니다.
>
> Enum 은 [상수](./constants.md) 문서에서 다루고 있으니 이 또한 참고 바랍니다.

## Union Type

관련된 여러 다른 값의 문자열을 묶어서 하나의 타입 처럼 사용 하는 경우입니다.

이들은 `유니온 타입`이라 부르며 그 값의 형태는 문자열만 가능합니다.

이 때 접미어는 `Type` 으로 사용합니다.

```ts
type KakaoCharacterNameType = 'jordy' | 'ryan' | 'muzi';
```

이들 타입은 간단히 정의하여 enum 과 유사한 목적으로 쓰일 수 있으며
특히 UI Component 의 props 에서 이용할 때 vscode 기준, 자동완성을 유도하여 매우 간편하게 코딩할 수 있도록 도와줍니다.

그러나 3개 이상 개수가 많아지거나 각 상수별로 주석을 두어 히스토리 관리를 하고자 한다면 `enum` 으로 정의하여 사용하길 권장합니다.

## Function Type

일반적으론 함수형을 별도로 만들어 쓸 일이 많진 않습니다.

그럼에도 특수하게 별도 함수형을 만들어 써야 할 경우 다음과 같이 `FnType` 이라는 접미어를 붙여 사용합니다.

```ts
type DateFormatFnType = (date: Date) => string;

const dateFormat: DateFormatFnType = (date) => {
  return date.toString();
};
```

## Generic Type

제네릭 타입명은 언어상에서 제약을 두지않습니다.

다만, 너무 여러가지 형태로 쓰일 경우, 코드 난독화의 원인이 되므로 기본적인 `T` 를 가급적 우선적으로 사용합니다.

그 외 2가지 이상이 쓰여야 하는 경우엔 다음 표를 참고하여 사용하시기 바랍니다.

| 작성 규칙 | 의미                | 주요 용도                                                                   |
| --------: | ------------------- | --------------------------------------------------------------------------- |
|         T | Type                | 가장 일반적으로 쓰이는 제네릭 타입명.<br/>특별한 문제가 없다면 이것만 사용. |
|         P | Props<br/>Parameter | Component Property 나 function parameter 용도                               |
|         R | Return Type         | 응답 결과                                                                   |
|         K | Key                 | Record 나 Map 등 key 가 필요한 곳에 쓰임                                    |
|         V | Value               | Record 나 Map 등 value 가 필요한 곳에 쓰임                                  |
|         S | State               | 상태값                                                                      |
|         E | Error               | 에러타입이 필요할 때 쓰임                                                   |

## 주의사항

TypeScript 의 type 문은 굉장히 광범위한 타입을 선언할 수 있기에 코드 난독화의 주범이 될 수 있습니다.

따라서 상기 언급한 것이 아닌 임의의 타입들은 가급적 충분히 고민한 뒤 적용하고 그 사유를 함께 공유하길 권장합니다.

그리고 자주 쓰이는 임의의 타입들이 있다면 그러한 패턴을 공유하여 함께 쓸 수 있도록 합니다.

## 규칙의 의의

type 선언에 필요한 만큼 제약을 걸음으로써 상호 코드 이해도를 높이고 혼란을 최소화 합니다.

타입명을 보았을 때 그것의 접미어만 보았을 때 그 타입의 대략적인 용도를 유추 할 수 있습니다.
